## План обновления монстров

### 1. Понять текущие источники данных по монстрам

- **Шаблоны монстров (процедурная генерация):**
  - Используется таблица `monster_templates` (модель `MonsterTemplate`) из файла `[src/waifu_bot/db/models/dungeon.py](src/waifu_bot/db/models/dungeon.py)`.
  - Базовые русские названия и вариации вида "Скелет-воин 1/2/3", "Живой доспех 1/2/3" сидятся в миграции `[alembic/versions/0006_seed_dungeon_content.py](alembic/versions/0006_seed_dungeon_content.py)` в списке `archetypes` и далее множатся в цикле по актам и вариантам.
- **Хардкод-данжи и их монстры:**
  - Сид-скрипт `[scripts/seed_dungeons.py](scripts/seed_dungeons.py)` содержит примерные данжи и список монстров для каждого (имена, статы, позиция), он пишет в таблицы `dungeons` и `monsters`.

### 2. Сформировать новый список монстров

- **Собрать/получить от вас новый список:**
  - Структура для процедурных шаблонов: базовые имена архетипов (например, "Скелет-воин", "Живой доспех", "Зомби-чемпион" и т.д.) и пожелания по тому, какие именно вариации ("-2", "-3") нужны.
  - Отдельный список для конкретных данжей (если там нужны строго фиксированные имена/композиции, отличающиеся от процедурной генерации).
- **Согласовать формат:**
  - Оговорить, будут ли в архетипах по-прежнему генерироваться суффиксы "1/2/3" автоматически, или вы хотите явные имена, которые уже включают уровень/ранг ("Скелет-воин-2").

### 3. Обновить процедурные шаблоны `monster_templates`

- **Изменение архетипов в миграции:**
  - Проанализировать блок `archetypes` в `[0006_seed_dungeon_content.py](alembic/versions/0006_seed_dungeon_content.py)`, где сейчас описаны базовые типы монстров с эмодзи, семейством, локациями и базовой сложностью.
  - На основе нового списка:
    - Обновить названия существующих записей в `archetypes` (например, заменить "Скелет-воин" на ваши новые имена, добавить/убрать типы вроде "Живой доспех").
    - При необходимости скорректировать `family`, `locs` (теги локаций), `base_diff`, `hp_bias`, `dmg_bias`.
  - Обсудить стратегию несовместимости с существующими данными в БД: так как миграции уже применены, для **боевого сервера** правильнее:
    - Создать **новую Alembic-миграцию**, которая через SQL `UPDATE`/`INSERT` изменит строки в таблице `monster_templates` (по текущим именам/ID), вместо правки исторической миграции `0006`.
- **Новая миграция для обновления шаблонов:**
  - Создать миграцию `alembic/versions/00xx_update_monster_templates_names.py`.
  - В ней:
    - Описать маппинг `старое_имя → новое_имя` (и при необходимости — новые поля `family`, теги и пр.).
    - Выполнить `UPDATE monster_templates SET ... WHERE name = :old_name` или по ID, если хотите 100% стабильность.
    - Для новых типов монстров — выполнить `INSERT` в `monster_templates` c выбранным диапазоном уровней/актов и базовыми статами.

### 4. Обновить хардкод-сид `scripts/seed_dungeons.py`

- **Коррекция списков монстров в примерах данжей:**
  - В `[scripts/seed_dungeons.py](scripts/seed_dungeons.py)` заменить имена монстров в полях `"name"` на новые (например, вместо "Скелет-Воин"/"Жрец Тьмы" использовать ваш новый нейминг вроде "Скелет-2", "Живой доспех-3", и т.д.).
  - При необходимости подправить `level`, `max_hp`, `damage`, `experience_reward`, `monster_type`, чтобы согласовать их с новой иерархией монстров.
- **Синхронизация с БД:**
  - Учесть, что сид-скрипт использует upsert-логику по `act` + `dungeon_number` и `position` монстра. Если скрипт планируется запускать поверх существующей базы, он сам обновит имена и статы у уже имеющихся монстров.
  - Если база в проде уже живёт своей жизнью (данжи менялись руками), можно сделать отдельную миграцию, которая аналогично `seed_dungeons.py` обновит конкретные записи в таблице `monsters`.

### 5. Стратегия применения и отката

- **Для разработки/стенда:**
  - Прогнать новую миграцию на локальной БД.
  - Запустить `scripts/seed_dungeons.py`, убедиться, что:
    - В таблице `monster_templates` появились/обновились нужные шаблоны.
    - В таблице `monsters` у сидовых данжей корректные имена и статы.
  - Протестировать пару реальных боёв (обычный данж, endless dungeon) и проверить, корректно ли отображаются новые названия.
- **Для продакшена:**
  - Применить только миграции (без повторного полного сида, если он может затереть живые данные), либо подготовить отдельную миграцию, которая точечно обновляет только те монстры/шаблоны, которые вы хотите переименовать.
  - Описать в комментарии к миграции, что она **idempotent** (не создаёт дубликатов и не ломает ссылки).

### 6. (Опционально) Документация и проверка совместимости

- Обновить при необходимости документацию по данжам и монстрам в файле `[docs/ENDLESS_DUNGEON_PLUS_SCHEMA.md](docs/ENDLESS_DUNGEON_PLUS_SCHEMA.md)`, если там упоминаются конкретные имена.
- Проверить, что новые имена не ломают формат сообщений бота (ограничения по длине строк, эмодзи, html-эскейпинг и т.п.).
