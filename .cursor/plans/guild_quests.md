Хорошая идея, отлично вписывается в механику игры. Покажу сначала концепцию, потом конкретику с примерами и схемой БД.Вот концепция системы со всеми деталями:

---

## Три слоя квестов

**Вехи (Milestones)** — главная механика. Не сбрасываются никогда, у каждой 3–5 тиров с нарастающими целями. Именно на них завязана основная прокачка гильдии, потому что они постоянно «тикают» в фоне пока участники играют в своё удовольствие. Прогресс никуда не девается, даже если гильдия долго неактивна.

**Ежедневные** — 3–4 квеста, сбрасываются в 00:00 МСК синхронно с таверной и слотами экспедиций. Дают немного Guild XP, основная функция — создать ощущение ежедневной точки сбора. Стоит добавить небольшой бонус лично участнику (например +5% EXP на сегодня), чтобы была мотивация.

**Еженедельные** — 2–3 квеста посложнее, дают солидный кусок Guild XP. Хорошо подходят для требований вроде «пройти N элитных монстров» или «отправить N видео» — что-то что за день нереально, но за неделю реально.

## Категории квестов и конкретные примеры

**Активность чата** — самая органичная категория для игры, потому что напрямую отражает реальное использование чата:

- *Стикер-марафон* I–V: 100 / 1 000 / 10 000 / 50 000 / 100 000 стикеров
- *Голос гильдии* I–III: 100 / 500 / 2 000 аудиосообщений (высокий XP за тир, т.к. аудио редкое)
- *Видеопоток* I–III: 50 / 300 / 1 500 видео
- *Гиф-армия* I–IV: 200 / 2 000 / 20 000 / 80 000 GIF
- *Тысяча слов* I–V: 1 000 / 10 000 / 100 000 / 500 000 / 2 000 000 текстовых

**Боевые** — отражают прогрессию:

- *Истребители монстров* I–V: 500 / 2 000 / 8 000 / 25 000 / 100 000 убитых
- *Охотники на боссов* I–IV: 20 / 100 / 500 / 2 000 боссов
- *Элитная бригада* I–III: 50 / 500 / 5 000 элитных монстров
- *Суммарный урон* I–V: 100k / 1M / 10M / 100M / 1B урона
- *Критический отряд* I–III: 500 / 5 000 / 50 000 крит-ударов

**Экспедиции** — отдельный счётчик, т.к. это не групповая активность в чате, а механика таверны:

- *Экспедиционный корпус* I–IV: 10 / 50 / 200 / 1 000 завершённых экспедиций
- *Без потерь* I–III: 10 / 50 / 200 экспедиций без смерти наёмниц
- *Долгий поход* I–III: суммарно 5ч / 50ч / 500ч времени в экспедициях

**Экономика**:

- *Золотой фонд* I–IV: 10k / 100k / 1M / 10M заработанного золота
- *Коллекционеры* I–III: 100 / 1 000 / 10 000 найденных предметов
- *Редкие вещи* I–III: 50 / 500 / 5 000 предметов редкости Rare и выше

## Система Guild XP и уровней

Предлагаю 20 уровней, чтобы было ощущение долгосрочной прокачки, но не уходить в бесконечность. Прогрессия XP нелинейная: первые уровни берутся быстро, дальше нарастает.

| Ур. | Guild XP | Разблокируется |
|-----|----------|----------------|
| 1 | 0 | Гильдия создана, базовый чат |
| 2 | 500 | +5% EXP для всех участников |
| 3 | 1 500 | Слот для гильдийской заметки (описание + флаг) |
| 5 | 4 000 | +5% золото с монстров для всех |
| 7 | 9 000 | +1 дополнительный слот экспедиции в день |
| 10 | 20 000 | +10% шанс выпадения предметов · Гильдийский магазин (уник. товары) |
| 12 | 35 000 | -10% стоимость найма наёмниц (действует для всех членов) |
| 15 | 60 000 | Гильдийское подземелье (особое, только для членов) |
| 18 | 100 000 | +15% EXP · Гильдийский банк (общий золотой фонд) |
| 20 | 150 000 | Гильдийский боссовый рейд · Уникальная рамка профиля |

Баффы накапливаются (ур. 10 даёт всё что было на ур. 2, 5, 7 тоже).

## Счётчик вклада внутри квеста

Каждый квест параллельно ведёт индивидуальный счётчик вклада участника. Это нужно для двух вещей: лидерборд «кто больше всего вложил» внутри карточки квеста (мотивация соревноваться), и личная награда при выполнении квеста. Логичнее всего делать личный бонус пропорциональным вкладу — например за Стикер-марафон III топ-3 вкладчика получают +X% EXP на 24ч.

## Схема БД

```sql
-- Шаблоны квестов (справочник)
CREATE TABLE guild_quest_templates (
  id              SERIAL PRIMARY KEY,
  type            VARCHAR(16) NOT NULL,  -- 'milestone' | 'daily' | 'weekly'
  category        VARCHAR(16) NOT NULL,  -- 'chat' | 'combat' | 'expedition' | 'economy'
  name            VARCHAR(128) NOT NULL,
  description     TEXT,
  metric          VARCHAR(64) NOT NULL,  -- 'stickers_sent' | 'monsters_killed' | ...
  reset_interval  VARCHAR(16),           -- NULL для milestone, 'daily', 'weekly'
  is_active       BOOLEAN DEFAULT TRUE
);

-- Тиры для вех (milestone_tiers)
CREATE TABLE guild_quest_tiers (
  id           SERIAL PRIMARY KEY,
  template_id  INT REFERENCES guild_quest_templates(id),
  tier         INT NOT NULL,             -- 1..5
  target_value BIGINT NOT NULL,
  reward_xp    INT NOT NULL,
  name_suffix  VARCHAR(32)               -- ' I', ' II' и т.д.
);

-- Состояние квеста для конкретной гильдии
CREATE TABLE guild_quests (
  id           SERIAL PRIMARY KEY,
  guild_id     INT NOT NULL,
  template_id  INT REFERENCES guild_quest_templates(id),
  tier_id      INT REFERENCES guild_quest_tiers(id),  -- NULL для daily/weekly
  current_val  BIGINT DEFAULT 0,
  completed_at TIMESTAMPTZ,
  expires_at   TIMESTAMPTZ,              -- NULL для milestone
  rewarded     BOOLEAN DEFAULT FALSE
);

-- Личный вклад участника в квест
CREATE TABLE guild_quest_contributions (
  quest_id    INT REFERENCES guild_quests(id),
  user_id     BIGINT NOT NULL,
  value       BIGINT DEFAULT 0,
  PRIMARY KEY (quest_id, user_id)
);
```

Обновление `current_val` и `contributions.value` происходит в том же транзакционном блоке, где уже считается урон монстру — никаких отдельных воркеров не нужно. Проверка завершения вехи — простой `IF current_val >= target_value AND completed_at IS NULL` после апдейта.

## Несколько нюансов баланса

Веса XP стоит привязать к редкости действия в игре. Аудио- и видеосообщения в реальных Telegram-чатах отправляют в 10–20 раз реже стикеров, поэтому XP за «Голос гильдии III» должен быть сопоставим со «Стикер-марафон V» даже при меньшем абсолютном числе. Иначе игроки просто проигнорируют аудио-вехи.

Для «Еженедельных» квестов стоит добавить механику голосования: раз в неделю гильдмастер или офицеры выбирают одну еженедельную цель из трёх предложенных — это создаёт внутригильдийную коммуникацию и ощущение собственного выбора.
