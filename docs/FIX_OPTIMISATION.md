## 🔴 Критические: убивают под нагрузкой

### 1. Горячий путь `group_message_damage` — слишком много работы на каждое сообщение

Каждое сообщение в любом групповом чате последовательно выполняет:
- `PlayerTelegramActivityMiddleware` → DB-запись (`last_active`, sync Telegram identity) на **каждый** апдейт
- `touch_player_chat_seen` + `touch_bot_group_chat_activity` → ещё два обращения к БД
- `chat_rewards.try_award` → Redis read/write
- `get_active_v1_cycle` → **отдельный SELECT к БД на каждое сообщение** (`gd_cycles WHERE status=active`)
- Условный write в Redis-буфер GD
- Guild GXP + war activity → commit в БД
- `CombatService.process_message_damage` → огромный граф: статы, аффиксы, legendary hooks, дропы, SSE publish
- `handle_abyss_attack` — вызывается всегда, даже если игрок не в Абиссе

В итоге при активном чате с 20 участниками — это легко 50–100 DB-сессий в минуту только на входящие сообщения. Запрос `get_active_v1_cycle` при этом никогда не кэшируется, хотя цикл меняет статус раз в 15–30 минут.

### 2. Двойной путь при активном GD + Solo (Dual-Path Hazard, §14.2)

Когда в чате активен `GDCycle`, каждое сообщение буферизуется для GD **и** при этом полностью выполняется ветка `solo combat + abyss`. Нагрузка удваивается именно в пиковые моменты — когда игроков больше всего активно в чате. Документ называет это проблемой, но решения не предлагает кроме ручного сброса через SQL.

### 3. LLM-вызовы в том же event loop, что и HTTP-сервер

Фоновые задачи `gd_v1_round` (каждые 20 сек), `expedition_tick` (каждые 30 сек) и `guild_war_narrative` (каждые 900 сек) вызывают OpenRouter из одного `asyncio` event loop, в котором работает FastAPI + aiogram. OpenRouter может отвечать 1–5+ секунд. Даже при `await` HTTP-запрос через `httpx` держит event loop занятым в фазах I/O, а в момент разбора ответа и записи в БД — блокирует его полностью. Все входящие WebApp-запросы в этот момент ждут.

Особенно опасен `gd_v1_round`: после LLM-вызова он делает массовую рассылку в Telegram всем участникам раунда — это пачка последовательных `Bot.send_message`, и каждый — сетевой вызов.

## 🟠 Серьёзные: деградация при росте аудитории

### 4. Монопроцессная архитектура без воркеров

Все компоненты в одном Python-процессе: FastAPI + aiogram + 9 фоновых `asyncio`-петель. Нет Celery, нет RQ, нет отдельных воркеров. Это значит:

- горизонтальное масштабирование невозможно без переработки (нельзя просто добавить процесс — будет гонка на фоновых задачах)
- падение процесса останавливает всё разом
- вся тяжёлая работа (LLM, массовые DM, flush в БД) соревнуется с обработкой WebApp-запросов за один event loop

### 5. Сохранение аудио из чата внутри обработчика сообщений

Шаг 2 в `group_message_damage`: `tavern_audio.save_chat_audio_from_message` — скачивает файл с серверов Telegram (внешний HTTP-запрос) прямо в обработчике. Даже при `async` это добавляет latency каждому аудиосообщению в чате и вытесняет другие задачи из event loop. Это должно быть fire-and-forget через `asyncio.create_task` или фоновый воркер.

### 6. `game_config` читается из БД без кэша

`game_config_service` хранит настраиваемые параметры (спам-лимиты, коэффициенты, стоимости) в таблице `game_config`. Эти значения используются в каждом combat-расчёте. Если они не кэшируются в памяти процесса или Redis — каждый бой делает дополнительные SELECT к БД. В документе кэширование этого слоя не упоминается.

### 7. `PlayerTelegramActivityMiddleware` — запись в БД на **каждый** апдейт

Middleware №1 в цепочке aiogram обновляет `Player.last_active` и синхронизирует Telegram-identity на абсолютно каждый входящий апдейт: и на сообщения, и на команды, и на callback-кнопки. При 10 активных группах по 30 сообщений в минуту — это 300 write-транзакций/мин только на `last_active`. Достаточно debounce-стратегии: обновлять не чаще раза в 5–10 минут через Redis TTL.

## 🟡 Умеренные: технический долг с потенциалом роста в проблему

### 8. Redis без подтверждённой durability для GD-буфера

GD-действия игроков (`gd_v1_buf:{cycle_id}`) живут только в Redis до обработки раунда. В документе нет упоминания о настройке AOF-персистентности или репликации. Потеря Redis = потеря всего прогресса текущего раунда группового подземелья. Для игрового состояния это особенно чувствительно.

### 9. SSE: fire-and-forget без восстановления состояния

`sse.publish_event` публикует события в Redis pub/sub канал `sse:{player_id}`. Если WebApp-клиент отключился (Telegram закрыли вкладку, ушёл в фон — типичное поведение на мобиле), события теряются. Клиент обязан делать полный refetch при восстановлении соединения. Пропущенный SSE о конце боя или завершении экспедиции = игрок не видит результат до следующего ручного открытия страницы.

### 10. Задержка chat_rewards до 60 секунд

Буфер наград (золото, опыт за активность в чате) сбрасывается в БД раз в 60 секунд. Игрок видит пустой кошелёк почти минуту после активности. При этом лимиты и кулдауны уже применены через Redis — визуальная несогласованность.

### 11. Armory full profile — тяжёлые джойны без кэша

`armory_service` описан как «wide joins (admin only, but heavy)» — агрегирует данные из инвентаря, статов, навыков, истории подземелий, экспедиций, гильдии. Кэширование не упоминается. Пока admin-only, но если появится публичный профиль с теми же джойнами — это сразу станет проблемой.

### 12. 189 зарегистрированных роутов и дублирование expedition

`expedition_routes.py` определяет 6 роутов, но **не подключён** через `include_router`. Живые маршруты экспедиций дублированы в `routes.py`. Это не прямая проблема производительности, но создаёт риск расхождения логики и усложняет поддержку. При быстром росте кодовой базы такие «мёртвые» модули начинают вводить в заблуждение.

## Итоговые приоритеты исправлений

Если ранжировать по соотношению усилий к эффекту:

**Быстрые wins:** закэшировать `get_active_v1_cycle` в Redis с TTL 30–60 сек; добавить debounce к `PlayerTelegramActivityMiddleware` через Redis; вынести `save_chat_audio` в `asyncio.create_task`.

**Среднесрочно:** вынести LLM-вызовы фоновых задач (`gd_v1_round`, `expedition_tick`) в отдельные `asyncio.Task` с изоляцией от основного цикла обработки сообщений, либо мигрировать на Celery/RQ воркеры.

**Архитектурно:** пока аудитория мала — система справится. При 500+ одновременно активных игроков монопроцессная архитектура упрётся в потолок event loop из-за конкуренции LLM + массовых DM + combat-вычислений.
