На основе архитектуры из документа видно, что сейчас бот построен по принципу **"монолит + единый event-loop"**:

* FastAPI
* Aiogram
* PostgreSQL
* Redis
* 9 фоновых циклов asyncio
* LLM-интеграции
* WebApp
* Armory

всё работает внутри одного Python-процесса. Это сильно упрощает разработку, но именно это становится главным ограничителем производительности. 

# Главные узкие места текущей архитектуры

## 1. Один процесс обслуживает всё

Сейчас один процесс одновременно:

* принимает Telegram webhook
* обрабатывает сообщения чатов
* крутит 9 фоновых циклов
* обслуживает WebApp API
* обслуживает SSE
* выполняет LLM-запросы
* генерирует нарративы

Если какой-нибудь `gd_v1_round` или `expedition_tick` зависнет на OpenRouter 10-15 секунд, то event-loop начинает испытывать нагрузку. 

---

## 2. Самый горячий участок — group_message_damage

Каждое сообщение группы проходит цепочку:

```
Redis
↓
PostgreSQL
↓
Chat Rewards
↓
Guild XP
↓
Raid
↓
Solo Combat
↓
Abyss
↓
SSE
```

То есть одно сообщение игрока может генерировать десятки операций. 

Для RPG, где онлайн может вырасти до нескольких тысяч пользователей, это станет первым потолком.

---

## 3. Нет очередей задач

В документе прямо указано:

> No Celery — all scheduling is asyncio.create_task + while True loops. 

То есть:

* нет распределения нагрузки
* нет отдельного воркера
* нет гарантированной доставки задач
* нет масштабирования по типу работы

---

## 4. Слишком много прямых обращений к PostgreSQL

Практически все сервисы работают напрямую через SQLAlchemy.

Нет:

* CQRS
* read replicas
* materialized views
* отдельного read cache

Из-за этого PostgreSQL станет главным ограничителем раньше Redis.

# Вариант 1. Минимальная модернизация (самый выгодный)

Практически без переписывания логики.

## Что меняем

### Выносим фоновые задачи

Сейчас:

```
FastAPI
 └─ asyncio loops
```

Станет:

```
FastAPI
Bot
Worker
```

через Celery или Dramatiq.

Я бы рекомендовал:

* Dramatiq + Redis

или

* Celery + Redis

---

### Что переносим

* gd_v1_round
* expedition_tick
* expedition_notify
* guild_war_narrative
* abyss_reset

---

### Результат

Обычно даёт:

* API быстрее на 20-40%
* Telegram реакции быстрее на 30-50%
* меньше лагов при генерации LLM

# Вариант 2. Разделение на сервисы

Следующий уровень.

## Схема

```
Gateway
│
├── Bot Service
├── WebApp API
├── Combat Service
├── Guild Service
├── LLM Service
└── Background Workers
```

Через:

* Redis Streams
  или
* RabbitMQ

---

## Что даёт

Сообщение игрока:

```
Telegram
↓
Bot Service
↓
Queue
↓
Combat Service
```

Бот отвечает мгновенно.

Урон считается отдельно.

---

Для вашей игры это уже позволяет держать десятки тысяч сообщений в час.

# Вариант 3. Игровой Event Bus

Если проект станет большим.

## Вместо

```
message -> combat -> guild -> abyss
```

делаем:

```
PlayerMessageEvent
```

и подписчики:

```
Combat
Guild
Raid
Abyss
Rewards
Statistics
```

---

Например:

Игрок написал сообщение.

В шину летит:

```json
{
  "type": "PlayerMessage"
}
```

А дальше каждый сервис решает сам:

* начислять награду
* бить босса
* качать гильдию
* считать статистику

---

Плюсы

Очень хорошо масштабируется.

Минусы

Потребует серьёзной переработки архитектуры.

# Вариант 4. Полный cloud-native

Для онлайна 50k+ игроков.

## Заменяем

### FastAPI

оставляем.

---

### Redis

на кластерный Redis:

* Redis Cluster
* KeyDB
* Dragonfly

Я бы выбрал Dragonfly.

Он часто даёт 2–5x прирост производительности без изменения кода.

---

### PostgreSQL

Добавляем:

```
PgBouncer
```

обязательно.

Сейчас его в архитектуре нет. 

Это обычно даёт огромный выигрыш для FastAPI.

---

Добавляем:

```
Primary PostgreSQL
    ↓
Read Replica
```

---

### Очереди

Redis Queue → RabbitMQ или Kafka

---

### Контейнеризация

Сейчас Docker-инфраструктуры в репозитории нет. 

Переводим на:

* Docker
* Kubernetes

# Вариант 5. Максимальная производительность

Если цель — очень высокий онлайн.

## Переписываем hot path

Сейчас:

```
Python
↓
SQLAlchemy
↓
PostgreSQL
```

---

Для боёв:

```
Redis
↓
Lua
↓
Dragonfly
```

или

```
Go Combat Service
```

---

Combat становится in-memory.

В PostgreSQL пишутся только результаты.

Так работают многие крупные idle/MMO проекты.

# Что бы я сделал именно для Waifu Reborn

Судя по описанию проекта, сейчас оптимальный путь выглядит так:

### Этап 1 (1–2 недели)

* PgBouncer
* индексы PostgreSQL
* вынести фоновые циклы в Dramatiq
* вынести LLM-задачи в отдельный worker

Ожидаемый выигрыш: **x1.5–x3**

---

### Этап 2 (2–4 недели)

* Combat Service отдельно
* Bot Service отдельно
* WebApp отдельно
* Redis Streams между ними

Ожидаемый выигрыш: **x3–x10**

---

### Этап 3 (при большом онлайне)

* Dragonfly вместо Redis
* PostgreSQL Replica
* Kubernetes
* Event-driven архитектура

Ожидаемый выигрыш: **x10–x50** относительно текущей схемы.

---

Если смотреть именно на документ, то самое большое влияние на быстродействие сейчас окажут не замены FastAPI или Aiogram, а:

1. вынос фоновых задач из общего процесса;
2. отделение LLM-генерации от игрового цикла;
3. разгрузка `group_message_damage`;
4. добавление PgBouncer;
5. переход от прямых вызовов сервисов к очередям событий для чатов с высокой активностью.
