# Анализ узких мест производительности HTML-страниц

## 1. Профиль (`profile.html`) — самая тяжёлая страница

**Инвентарь (вкладка 1.2)**
- 2D пиксельная модель персонажа генерируется динамически и обновляется при каждой смене экипировки — если это canvas/SVG-рендеринг на клиенте, каждое обновление может вызывать перерисовку
- Сетка сумки до 72 ячеек с иконками предметов, рамками редкости и оверлей-индикаторами — при рендере всех ячеек разом без виртуализации браузер создаёт сотни DOM-нод одновременно
- Фильтр и сортировка сумки работают на всём массиве предметов без пагинации по умолчанию — при 72 ячейках и переключении фильтров каждый раз перестраивается весь список

**Подробная информация (вкладка 1.3)**
- 6 аккордеонов с формулами характеристик и 7-й контейнер со статистикой — всё это требует отдельных запросов к API или одного тяжёлого агрегирующего запроса
- Формула регенерации HP (`HP_макс × (1 − e^(−ВЫН/100))`) и Торговли пересчитываются на клиенте при каждом открытии вкладки
- Блок «Статистика» подтягивает агрегаты за всё время игры (суммарный урон, золото и т.д.) — без кэширования это тяжёлые COUNT/SUM-запросы к БД

## 2. Подземелья (`dungeons.html`)

**Список подземелий**
- Все подземелья всех актов подгружаются при открытии страницы, даже свёрнутые акты — нет ленивой загрузки по акту
- Карточки подземелий включают статус прохождения, уровни монстров, тип локации — несколько JOIN-запросов к таблицам подземелий, монстров и прогресса игрока

**Карточка активного боя**
- Изображения монстров (WebP) с fallback-цепочкой из 4 уровней — каждый сбой `onerror` инициирует новый HTTP-запрос; при отсутствии индивидуального изображения браузер делает 3-4 последовательных запроса
- Real-time обновление прогресс-баров HP монстра и ОВ — если реализовано через polling, а не WebSocket, частые запросы к серверу дополнительно нагружают соединение внутри WebApp Telegram

**Экспедиции**
- Таймер обратного отсчёта `ЧЧ:ММ:СС` обновляется в реальном времени — при реализации через `setInterval` + запросы к серверу для синхронизации создаёт постоянный фоновый трафик
- Превью отряда с «оценкой шанса успеха» пересчитывается динамически при каждом изменении состава или длительности — если расчёт идёт на сервере, каждое изменение слайдера/чекбокса порождает отдельный запрос

## 3. Таверна (`tavern.html`)

- Вкладка «Найм»: пул из 5 наёмниц с процедурной генерацией аватаров, перками, статами — при первой загрузке страницы запрашивается весь пул целиком
- Вкладка «Прокачка»: стоимость прокачки каждой наёмницы пересчитывается формульно при рендере — если не кэшировано, 10 наёмниц × формула = 10 вычислений на рендер
- Все 4 вкладки, вероятно, подгружают данные при инициализации страницы, а не лениво по клику

## 4. Общие структурные проблемы

**Основной чердак (ОЧ) на всех страницах**
- ОЧ присутствует на всех 9 страницах и содержит: уровень+EXP, активное подземелье, статус экспедиций, золото, энергию — это 3–5 разных источников данных, каждый из которых может быть отдельным API-запросом при каждом переходе между страницами
- Если ОЧ не реализован как единый глобальный стейт (например, Redux/Zustand в SPA), данные запрашиваются повторно при каждом переключении страницы

**Изображения**
- Портрет ОВ (PNG/WEBP, соотношение 3:2) без указания размеров в HTML вызывает reflow при загрузке
- Иконки предметов в сетке сумки (до 72 штук) без lazy loading загружаются все сразу
- 2D пиксельная модель персонажа в слотах экипировки — если это составное изображение из слоёв (базовая модель + каждый предмет), число HTTP-запросов пропорционально количеству экипированных предметов

**База данных**
- Запрос данных профиля ОВ при загрузке `profile.html` потенциально затрагивает: таблицу персонажа, инвентарь (все предметы), характеристики, навыки, статистику — без правильных индексов и кэширования это N+1 проблема
- Таблица `monster_affixes` с JSON-полями (`behavior_params`, `incompatible_with`, `allowed_families`) — JSON-поиск в PostgreSQL без индексов `GIN` значительно медленнее обычных полей
- Агрегаты статистики (`суммарный урон`, `заработано золота`) без материализованных представлений считаются полным сканом таблиц событий

**Telegram WebApp-специфика**
- WebApp открывается внутри встроенного браузера Telegram (WKWebView/Android WebView) — JS-движок медленнее полноценного браузера, тяжёлые вычисления на клиенте (формулы характеристик, расчёт урона) будут заметно тормозить
- Отсутствие Service Worker для кэширования статики означает, что иконки предметов, спрайты, фоны загружаются заново при каждом открытии WebApp
- Холодный старт WebApp: если все JS-бандлы не разбиты на чанки (code splitting), пользователь ждёт загрузки всего приложения даже для открытия одной страницы

**Магазин (`shop.html`) и механика Gamble**
- Ассортимент магазина меняется по актам и включает предметы с рассчитанными ценами (формула Торговли для каждого предмета) — при большом ассортименте расчёт цен для всего списка на рендере создаёт задержку
- Функция Gamble, если реализована с анимацией, требует быстрого JS-цикла — в WebView может давать видимые фризы

## Приоритизация по критичности

| Приоритет | Проблема | Страница |
|---|---|---|
| 🔴 Критично | Загрузка всех данных сразу без lazy/pagination | Все |
| 🔴 Критично | Повторные запросы данных ОЧ при каждом переходе | Все |
| 🔴 Критично | Fallback-цепочка из 4 последовательных HTTP-запросов изображений | dungeons.html |
| 🟠 Высокий | N+1 запросы при рендере инвентаря | profile.html |
| 🟠 Высокий | Отсутствие Service Worker / кэша статики | Все |
| 🟠 Высокий | JSON-поля без GIN-индексов в monster_affixes | dungeons.html |
| 🟡 Средний | Отсутствие code splitting JS-бандла | Все |
| 🟡 Средний | Агрегаты статистики без материализованных представлений | profile.html |
| 🟡 Средний | Динамический пересчёт формул при каждом рендере без мемоизации | profile.html, shop.html |
| 🟢 Низкий | Отсутствие размеров у изображений (reflow) | profile.html |
